# Тестовое задание: Ticket Workspace

## Кратко

Нужно доработать рабочее место оператора поддержки в существующем React-приложении.

Фокус задания не в количестве кода и не в визуальной полировке. Нам важно увидеть, как вы читаете существующий проект, проводите границы ответственности, работаете с route state, browser state и API/cache state, а также проверяете поведение тестами.


Использование AI разрешено и приветствуется. При этом вы должны быть готовы объяснить каждую существенную часть итогового решения: где лежит state, почему выбраны такие границы, какие edge cases покрыты и как работают тесты.

## Бизнес-сценарий

Есть рабочее место оператора поддержки.

Оператор открывает карточки заявок, быстро возвращается к списку и помечает заявки как проверенные. В списке ему нужно видеть последние открытые заявки, чтобы быстро вернуться к недавнему контексту.

В стартовом проекте уже есть:

- список заявок `/tickets`;
- карточка заявки `/tickets/:ticketId`;
- 10 моковых заявок;
- mock RTK Query API;
- command palette по `Ctrl/⌘ K`;
- заготовки для workspace command registry и recent ticket storage.

## Что нужно реализовать

### 1. Команды в карточке заявки

В контексте карточки заявки добавьте команды в command palette:

- `Open ticket list` - перейти на `/tickets`;
- `Mark current ticket as reviewed` - пометить текущую заявку как `reviewed` через существующую mutation.

Команды, которые зависят от текущей заявки, должны быть доступны только когда пользователь находится на странице заявки.

### 2. Последние открытые заявки

В списке заявок отобразите блок с 3 последними открытыми заявками.

Требования:

- заявка считается открытой, когда пользователь попал на `/tickets/:ticketId`;
- последние заявки нужно хранить в `localStorage`;
- порядок отображения: самая свежая заявка первая;
- повторное открытие заявки поднимает ее наверх;
- дублей быть не должно;
- если id из storage больше нет в текущем списке заявок, такую запись нужно скрыть.

### 3. Edge cases для recent tickets

Обработайте случаи:

- `localStorage` недоступен;
- значение пустое;
- значение повреждено или не того типа;
- id из storage больше нет в текущем API-ответе.

### 4. Архитектурные границы

Сохраните границы существующего проекта:

- `src/app/command-palette` остается generic runtime;
- command palette не должна импортировать ticket API, ticket types или ticket business logic;
- ticket-aware команды собираются во workspace/feature слое;
- текущая заявка определяется из URL;
- не нужно заводить отдельный глобальный `currentTicketId`, который дублирует route state.

### 5. Тесты

Добавьте минимальное покрытие:

- unit test на recent ticket storage;
- integration test на command palette: открыть с клавиатуры, выполнить `Mark current ticket as reviewed`, проверить изменение статуса;
- integration или component test на отображение последних 3 заявок в списке.

## Что нужно сдать

Кроме кода, заполните два файла:

- `SOLUTION.md`
- `AI_USAGE.md`

В `SOLUTION.md` опишите:

- куда положили state и почему;
- где проходит граница command palette / ticket feature;
- какие edge cases покрыты;
- какие trade-offs приняли;
- что сделали бы следующим PR.

В `AI_USAGE.md` опишите:

- какие AI-инструменты использовали;
- какие задачи отдавали AI;
- что проверяли вручную;
- где AI ошибся или где вы скорректировали его решение.

## Проверка

Перед сдачей убедитесь, что проходят:

```bash
npm test
npm run build
```

Если какая-то проверка не проходит, укажите это в `SOLUTION.md` и объясните причину.

## Что не нужно делать

- Не подключать реальный backend.
- Не делать авторизацию.
- Не переписывать приложение целиком.
- Не менять стек проекта.
- Не строить полноценную дизайн-систему.
- Не добавлять сложные фичи вне задачи.

## На что мы смотрим

Мы оцениваем не только итоговое поведение, но и ход инженерных решений:

- умеете ли вы использовать существующие слои проекта;
- не протекает ли ticket business logic в generic app runtime;
- корректно ли разделены URL state, browser state и API/cache state;
- покрыты ли важные edge cases;
- можно ли по тестам и документации понять, почему решение корректно.
